# GSD-2 项目核心模块与工作流分析

## 一、项目概述

GSD-2（Get Shit Done 2）是原始 GSD 提示框架的进化版本，从一个简单的 Markdown 提示集合转变为真正的独立编码代理。该项目拥有 1,900 颗星标和 163 个分支，基于 Pi SDK 构建，提供对代理工具本身的直接 TypeScript 访问能力。

原始 GSD 作为 Claude Code 的提示框架而走红，但它在与工具的对抗中举步维艰——通过斜杠命令注入提示，希望 LLM 能够遵循指令，但实际上无法控制上下文窗口、会话或执行。GSD-2 的出现彻底改变了这一局面，它能够真正做到 v1 只能请求 LLM 去做的事情：在任务之间清除上下文、在调度时精确注入正确的文件、管理 Git 分支、跟踪成本和 Token、检测卡住循环、从崩溃中恢复、以及自动推进整个里程碑而无需人工干预。

GSD-2 的核心价值主张是：「一个命令。离开。回来时是一个具有干净 Git 历史的项目。」

## 二、核心设计理念

### 2.1 状态存储于磁盘

GSD-2 最核心的设计原则是「状态存储于磁盘」。`.gsd/` 目录是唯一的真实来源。自动模式读取它、写入它，并根据其内容推进工作。没有内存状态在会话之间存活。这使得崩溃恢复、多终端 steering 和会话恢复成为可能。

这种设计解决了传统 AI 助手面临的最大问题之一：在长时间运行的过程中，AI 的上下文会积累「垃圾」，导致质量下降。通过将状态持久化到磁盘，GSD-2 可以在每个任务开始时提供一个干净的上下文窗口。

### 2.2 每单元新鲜会话

GSD-2 的另一个核心原则是「每单元新鲜会话」。每次调度都会创建一个新的代理会话。LLM 开始时只有一个干净的 200k Token 上下文窗口，其中只包含预先内联的必要文件。这防止了上下文积累导致的质量下降。

传统的 AI 辅助开发中，AI 会在一个会话中持续工作，随着上下文窗口的填充，质量会逐渐下降。GSD-2 通过在每个任务、研究阶段、规划步骤获取干净的上下文窗口来解决这个问题。

### 2.3 自主执行模式

GSD-2 的自动模式（`/gsd auto`）是其与传统提示框架的根本区别。用户运行命令后可以离开，AI 会自动研究、规划、执行、验证、提交并推进到每个切片，直到里程碑完成。这不是 LLM 自己调用自己的循环（v1 的「自动模式」），而是一个由磁盘上的文件驱动的状态机。

## 三、工作流层次结构

GSD-2 将工作结构化为一个层次体系：

```
里程碑（Milestone）→ 可发运的版本（4-10 个切片）
切片（Slice）→ 一个可演示的垂直能力（1-7 个任务）
任务（Task）→ 一个适合上下文窗口的工作单元
```

铁律是：**任务必须适合一个上下文窗口。** 如果不能，那就是两个任务。

## 四、核心技能模块分类

### 4.1 工作流引擎

**/gsd** 是分步模式——每次执行一个单元，在每个单元之间暂停，向导显示完成的内容和下一步。相同的自动机，但用户保持在循环中。没有项目？它开始讨论流程。路线图存在？它规划或执行下一步。

**/gsd auto** 是自主模式——研究、规划、执行、验证、提交、重复。每个任务都有新鲜的上下文窗口，无需值守。

**/gsd next** 是分步模式的显式别名。

### 4.2 项目管理

**/gsd discuss** 用于讨论架构和决策（可与自动模式并行工作）。**/gsd status** 显示进度仪表板。**/gsd queue** 将未来的里程碑加入队列（自动模式期间安全）。

### 4.3 验证与审查

**verification_commands** 是可配置的 shell 命令（如 `npm run lint`、`npm run test`），在任务执行后自动运行。失败会触发自动修复重试，然后才继续。验证层级包括：静态检查 → 命令执行 → 行为测试 → 人工审查（仅当代理确实无法自行验证时）。

### 4.4 Git 策略

GSD-2 使用分支-per-切片加压缩合并的 Git 策略。每个里程碑在 `milestone/<MID>` 分支的自己的 Git worktree 中运行。所有切片工作按顺序提交——无分支切换、无合并冲突。当里程碑完成时，它被压缩合并到 main 作为一次干净的提交。

### 4.5 成本管理

GSD-2 提供完整的成本管理功能：每个单元的 Token 使用量和成本都被捕获，按阶段、切片和模型细分。仪表板显示运行总数和预测。预算上限可以在超支前暂停自动模式。

### 4.6 技能系统

GSD-2 包含 15 个内置技能，涵盖前端设计、SwiftUI、Rust、Axum、Tauri、GitHub Actions、安全审计、代码审查、测试和 lint 等领域。

技能发现模式包括：`auto`（自动发现和应用技能）、`suggest`（识别但需要确认，默认值）、`off`（无技能发现）。

技能生命周期管理包括：技能遥测（跟踪每个自动模式单元中可用的技能和加载的技能）、技能健康仪表板（查看技能表现，标记成功率低于 70%、Token 使用上升 20%+、或超过配置天数未使用的技能）、陈旧检测（自动取消优先级排序未使用的技能）。

## 五、技术架构特点

### 5.1 基于 Pi SDK

GSD-2 是构建在 Pi SDK 之上的 TypeScript 应用程序。它嵌入 Pi 编码代理并使用 GSD 工作流引擎、自动模式状态机和项目管理原语对其进行扩展。

### 5.2 调度管道

自动模式的调度管道包括以下步骤：读取磁盘状态（STATE.md、路线图、计划）；确定下一个单元类型和 ID；分类复杂度 → 选择模型层级；应用预算压力调整；检查路由历史以进行自适应调整；动态模型路由（如果启用）→ 为层级选择最便宜的模型；解析有效模型（带回退）；检查待处理的捕获 → 如需要则分类；构建调度提示（应用内联级别压缩）；创建新的代理会话；注入提示并让 LLM 执行；完成时：快照指标、验证工件、持久化状态；循环到步骤 1。

### 5.3 原生引擎

性能关键操作使用 Rust N-API 引擎：grep（ripgrep 支持的内容搜索）、glob（gitignore 感知的文件发现）、ps（跨平台进程树管理）、highlight（基于 syntect 的语法高亮）、ast（通过 ast-grep 进行结构化代码搜索）、diff（模糊文本匹配和统一 diff 生成）、ANSI 感知的文本测量和换行、HTML 到 Markdown 的转换、图像解码/编码/调整大小、模糊文件路径发现、本机剪贴板访问、libgit2 支持的 Git 读取操作。

### 5.4 捆绑扩展

GSD-2 捆绑了 12 个支持扩展，包括：GSD（核心工作流引擎）、Browser Tools（基于 Playwright 的浏览器自动化）、Search the Web（Brave Search、Tavily 或 Jina 页面提取）、Google Search（Gemini 支持的网络搜索）、Context7（最新的库/框架文档）、Background Shell（具有就绪检测的长时间运行进程管理）、Subagent（具有隔离上下文窗口的委托任务）、Mac Tools（通过辅助功能 API 的 macOS 原生应用自动化）、MCPorter（按需延迟 MCP 服务器集成）、Voice（macOS、Linux 的实时语音转文本）、Slash Commands（自定义命令创建）、LSP（语言服务器协议——诊断、定义、引用、悬停、重命名）。

### 5.5 捆绑代理

GSD-2 捆绑了三个专用代理：Scout（快速代码库重建——用于交接的压缩上下文）、Researcher（网络研究——查找和综合最新信息）、Worker（通用用途——在隔离的上下文窗口中执行）。

## 六、与 Superpowers 和 gstack 的对比分析

### 6.1 架构模式

GSD-2 与 Superpowers 和 gstack 最根本的区别在于其架构模式。Superpowers 是一个技能框架，通过自动触发机制确保工作流纪律；gstack 是一个命令集合，通过显式斜杠命令提供专业角色能力；GSD-2 则是一个完整的代理应用程序，基于 Pi SDK 构建，能够直接控制代理会话的各个方面。

GSD-2 不依赖于向 LLM 注入提示来「希望」它做正确的事情，而是通过代码直接控制上下文管理、会话生命周期、Git 操作和成本跟踪。这种架构使得 GSD-2 能够实现真正的自动化——包括崩溃恢复、自适应重新规划、自动里程碑验证等高级功能。

### 6.2 状态管理

GSD-2 的「状态存储于磁盘」原则是其区别于其他系统的关键特征。Superpowers 和 gstack 主要依赖于会话内的状态和上下文，而 GSD-2 将所有状态持久化到 `.gsd/` 目录，使得：崩溃恢复成为可能——即使会话死亡，下一个 `/gsd auto` 可以读取幸存的状态文件并恢复；多终端 steering 成为可能——两个终端可以同时操作同一个项目，决策在下一个阶段边界自动被采用；会话恢复成为可能——可以从中断的地方继续。

### 6.3 自主能力

GSD-2 的自动模式（`/gsd auto`）代表了与 Superpowers 和 gstack 不同的自动化水平。Superpowers 通过技能触发来确保流程纪律，但仍然需要用户在循环中；gstack 提供了强大的命令集，但仍然需要用户显式调用；GSD-2 的自动模式可以完全自主运行，用户可以「设置后离开」，回来时项目已经构建完成。

### 6.4 成本控制

GSD-2 是三者中唯一提供完整成本管理功能的系统。它追踪每个单元的 Token 使用量和成本，按阶段、切片和模型细分，提供预算上限控制，允许在超支前暂停自动模式。这种精细的成本控制对于大规模项目或商业使用场景尤为重要。

### 6.5 工作流结构

Superpowers 和 gstack 的工作流结构相对扁平——用户根据需要调用不同的技能或命令。GSD-2 则建立了严格的层次结构：Milestone（里程碑）→ Slice（切片）→ Task（任务），每个层次都有明确的定义和转换规则。这种结构确保了工作的可预测性和可管理性。

| 特性 | Superpowers | gstack | GSD-2 |
|------|-------------|--------|-------|
| 架构类型 | 技能框架 | 命令集合 | 独立应用 |
| 触发机制 | 自动触发 | 显式命令 | 状态机驱动 |
| 状态管理 | 会话内 | 会话内 | 磁盘持久化 |
| 自主程度 | 需要用户在环 | 需要用户在环 | 可完全自主 |
| 成本控制 | 无 | 有限 | 完整 |
| 工作流结构 | 扁平 | 扁平 | 层次化 |
| 崩溃恢复 | 无 | 无 | 有 |
| Git 策略 | 基本 | 基本 | 完整 worktree |

## 七、可复用模式提炼

### 7.1 状态机驱动模式

GSD-2 的自动模式是由磁盘上的文件驱动的状态机。系统读取 `.gsd/STATE.md`，确定下一个工作单元，创建一个新的代理会话，注入一个包含所有相关上下文预先内联的专注提示，然后让 LLM 执行。这种「状态存储于磁盘」的模式使得自主执行、崩溃恢复和多终端协调成为可能。

### 7.2 上下文预加载模式

GSD-2 的调度提示包括内联的任务计划、切片计划、之前的任务摘要、依赖项摘要、路线图摘录和决策注册表。LLM 从它需要的一切开始，而不是花费工具调用来读取文件。这种模式被称为「上下文预加载」，它消除了 AI 在无关的代码探索上浪费的 Token。

### 7.3 验证强制执行模式

GSD-2 的验证系统采用「must-haves」——机械可检查的结果，包括：Truths（可观察的行为）、Artifacts（必须存在真实实现而非存根的文件）、Key Links（工件之间的导入和连接）。验证层级从静态检查到命令执行、行为测试，最终到人工审查。

### 7.4 复杂度分类路由

GSD-2 根据单元复杂度选择模型层级：light（轻量级）→ standard（标准）→ heavy（重量级）。系统还支持动态模型路由，根据成本感知选择最便宜的模型。这种模式在质量和成本之间提供了灵活的平衡。

### 7.5 技能生命周期管理

GSD-2 提供完整的技能生命周期管理：技能遥测（跟踪技能使用情况）、技能健康仪表板（显示成功率、Token 趋势、陈旧技能）、陈旧检测（自动取消优先级排序未使用的技能）、Heal-Skill（分析代理是否偏离技能指令并提出修复建议）。

### 7.6 双终端工作模式

GSD-2 的「两个终端，一个项目」工作模式允许用户在两个终端中同时工作：一个运行自动模式进行构建，另一个进行 steering、讨论和状态检查。两个终端读取和写入相同的 `.gsd/` 文件，终端 2 的决策在下一个阶段边界自动被终端 1 采用。

### 7.7 里程碑验证门控

GSD-2 在所有切片完成后运行一个 `validate-milestone` 门控，比较路线图成功标准与实际结果，然后才封存里程碑。这种机制确保了只有真正满足需求的代码才会被交付。

## 九、核心模块说明

GSD-2 的关键模块包括：`auto.ts`（自动模式状态机和编排）、`auto-dispatch.ts`（声明式调度表）、`auto-prompts.ts`（带内联级别压缩的提示构建器）、`auto-worktree.ts`（worktree 生命周期）、`complexity-classifier.ts`（单元复杂度分类）、`model-router.ts`（带成本感知的动态模型路由）、`routing-history.ts`（自适应学习）、`captures.ts`（火后忘记的思维捕获和分类）、`triage-resolution.ts`（捕获解决方案）、`visualizer-overlay.ts`（工作流可视化 TUI 覆盖）、`metrics.ts`（Token 和成本跟踪账本）、`state.ts`（从磁盘派生状态）、`preferences.ts`（偏好加载、合并、验证）、`git-service.ts`（Git 操作）、`memory-extractor.ts`（从会话记录中提取可重用知识）、`memory-store.ts`（跨会话持久化内存存储）。

## 十、总结与启示

GSD-2 提供了一种与 Superpowers 和 gstack 根本不同的 AI 驱动开发方法论。它不仅是一个提示框架或命令集合，而是一个完整的代理应用程序，能够直接控制代理会话的各个方面。

其核心创新包括：状态存储于磁盘的设计使得真正的自主执行和崩溃恢复成为可能；每单元新鲜会话确保了持续的代码质量；完整的工作流层次结构提供了可预测性和可管理性；精细的成本控制支持大规模商业使用场景；状态机驱动的自动模式实现了「设置后离开」的工作流程。

这三种方法论代表了 AI 驱动开发的三个不同方向：Superpowers 强调流程纪律，gstack 强调专业角色能力，GSD-2 强调系统控制。它们可以相互补充，为构建更完善的 AI 开发工作流提供有价值的参考。
